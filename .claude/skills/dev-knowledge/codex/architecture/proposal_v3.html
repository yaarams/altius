Context

Architecture

Wrapper-controlled delegation. The blue subgraph in the centre is the orchestration loop — one iteration per plan item, with think / analyze reflection points before, during, and after every Codex run.

Orchestration Loop — how I work alongside Codex

Codex is never a black box that swallows the whole task. Each codex exec is one step in my plan; I retain control before, during, and after every invocation.

For each plan item P that needs Codex help:
  think(P)                              ← reason about prompt + acceptance criteria
  write workspace/tmp/codex-prompts/P.md
  start = now(); spawn:
      bin/codex-run.sh <repo> <mode> <prompt> | tee events.jsonl
  while running:
      tail -F events.jsonl                ← stream JSONL
      every N events / T seconds:
          status = parse_jsonl(events.jsonl)
          think(status)                   ← progress reflection
          if user_visible_milestone: email/notion update
          if budget_exceeded or stuck: kill + analyze
  on exit:
      analyze(exit_code, last_message, audit)
      if exit==0 and audit==clean:
          xpcomplete_agent_plan_items([P.id])
          append note to task_queue
      else:
          replan: add follow-up plan item, log decision
final: review diff, run repo validation, commit + push + PR (me, not Codex)

Status checking pattern

The wrapper writes JSONL to workspace/tmp/codex-runs/<ts>/events.jsonl. I poll it from my parent loop:

# Cheap progress probe — last 3 events + line count
LOG=workspace/tmp/codex-runs/<ts>
wc -l "$LOG/events.jsonl"
tail -n 3 "$LOG/events.jsonl" | jq -r '.type + "  " + (.tool_call.name // .content // "")'

I feed the snapshot into a think step and reflect to the user via email when it’s a milestone (tests green, refactor done, blocker hit). The wrapper itself never talks to me — reflection is always the parent agent’s job.

Plan-tracking discipline

  • Each Codex exec = exactly one plan item.
  • xpcomplete_agent_plan_items only after exit 0, audit clean, and diff matches expected scope.
  • A failed run becomes a new plan item (“investigate Codex failure on P”) plus a queue note. Never silent retry.
  • Long-running tasks self-schedule a wake-up via xpschedule-create instead of burning a turn idle-watching.

Wrapper — bin/codex-run.sh

The skill’s enforcement core. Codex is never invoked directly; only via this wrapper.

#!/usr/bin/env bash
set -euo pipefail

# ---- args ----
REPO="$1"          # frontend | xpander-mono | xpander-sdk | docs
MODE="$2"          # read | write
PROMPT_FILE="$3"   # path to prompt; never inline so we audit prompts

# ---- canonicalize + allowlist ----
DEV_ROOT=/agent/data/dev
case "$REPO" in frontend|xpander-mono|xpander-sdk|docs);; *) exit 2;; esac
REPO_DIR="$DEV_ROOT/$REPO"
[[ -d "$REPO_DIR/.git" ]] || exit 2

case "$MODE" in
  read)  S=read-only       APPROVAL=never ;;
  write) S=workspace-write APPROVAL=never ;;
  *) exit 2 ;;
esac

TS=$(date -u +%Y%m%dT%H%M%SZ)
LOG="workspace/tmp/codex-runs/$TS"; mkdir -p "$LOG"; touch "$LOG/.start"

# ---- redact env ----
export HOME=/agent/data/.home
unset XPANDER_API_KEY XPANDER_AGENT_ID XPANDER_AGENT_TOKEN GH_TOKEN GITHUB_TOKEN
export OPENAI_API_KEY="${CODEX_OPENAI_API_KEY:?missing}"

# ---- run codex with locked flags ----
codex exec \
  --cd "$REPO_DIR" --sandbox "$S" --ask-for-approval "$APPROVAL" \
  --ignore-user-config --ignore-rules --ephemeral \
  --json -o "$LOG/last_message.txt" \
  -c sandbox_workspace_write.network_access=false \
  -c sandbox_workspace_write.writable_roots='[]' \
  -c shell_environment_policy.exclude='["XPANDER_*","GH_*","GITHUB_*","AWS_*"]' \
  - < "$PROMPT_FILE" | tee "$LOG/events.jsonl"

# ---- post-flight audit ----
VIO=$(find /agent/data/workspace /agent/data/.home/.codex \
        /agent/data/.home/.config /agent/data/.home/.gitconfig \
        -newer "$LOG/.start" ! -path "$LOG/*" 2>/dev/null | head -20)
[[ -z $VIO ]] || { echo "GUARDRAIL VIOLATION: $VIO" >&2; exit 99; }

git -C "$REPO_DIR" status --porcelain > "$LOG/git_status.txt"
for other in "$DEV_ROOT"/*/.git; do
  od="${other%/.git}"; [[ "$od" == "$REPO_DIR" ]] && continue
  [[ -z "$(git -C "$od" status --porcelain 2>/dev/null)" ]] || { echo "$od dirty" >&2; exit 99; }
done

echo "✅ codex-run complete. Logs: $LOG"

Skill outline — workspace/dev-knowledge/skills/codex_runner.md (not yet created)

When to use

  • Self-contained code edits with clear acceptance criteria.
  • Doesn’t need xpander-runtime-aware reasoning.
  • Doesn’t touch workspace/skills, memory, plans, or external services.
  • Examples: implement function X per spec, refactor file Y, write tests for Z, fix compile errors after dep bump.

When NOT to use

  • Anything touching skills / memory / queue / identity.
  • Cross-repo or multi-PR work.
  • Anything needing my Notion / email / CDN tools.
  • Anything needing a human signoff mid-task.

Pre-flight checklist

  • task_queue shows the right in_progress task.
  • Repo on the correct integration branch (MUST_READ.md §7).
  • CODEX_OPENAI_API_KEY is set.
  • Prompt file exists at workspace/tmp/codex-prompts/<id>.md.

Pinned system prompt

  • Operate inside the wrapper-controlled cwd. Don’t read/write outside.
  • Don’t commit/push/PR. Stop after producing the change.
  • Honor repo’s AGENTS.md; on conflict, this prompt wins.
  • Output a final summary listing every changed file.

v3 Update — Identity, Token Usage, Configurable Model

Three tightenings on top of v2:

1. Commits, branches, PRs are exclusively mine

Codex is a read+edit hand. It never runs git commit, git push, git switch -c, git tag, gh, or any auth’d HTTP that could open a PR; never writes to ~/.gitconfig or ~/.config/gh/**. Defense-in-depth:

LayerMechanism
Env redactionunset GH_TOKEN GITHUB_TOKEN GH_HOST GIT_AUTHOR_* GIT_COMMITTER_* + shell_environment_policy.exclude=["GH_*","GITHUB_*","GIT_*_NAME","GIT_*_EMAIL"]
Identity scrubGIT_CONFIG_GLOBAL=/dev/null + GIT_CONFIG_SYSTEM=/dev/null → any git commit Codex tries fails with please tell me who you are
Pinned prompthard line: “you MUST NOT run git commit, git push, git switch -c, git tag, gh, or any HTTP call to git/GitHub.”
Post-run auditcompare git rev-parse HEAD against pre-flight; HEAD movement → exit 99. Also ~/.config/gh/** + ~/.gitconfig newer-than-start → exit 99.
PR ownershipI review the working-tree diff, run repo validations, then git commit + git push + gh pr create from outside the wrapper, under the agent identity per MUST_READ.md §2a.

2. Token usage capture

Codex emits a turn.completed event with usage on every turn. The wrapper distills it into a per-run summary and an append-only ledger:

# turn.completed event
{"type":"turn.completed","usage":{"input_tokens":24763,"cached_input_tokens":24448,"output_tokens":122}}

# wrapper post-step
jq -s '
  [.[] | select(.type=="turn.completed") | .usage] as $turns
  | { runs: ($turns|length),
      input_tokens:        ($turns|map(.input_tokens)        |add // 0),
      cached_input_tokens: ($turns|map(.cached_input_tokens) |add // 0),
      output_tokens:       ($turns|map(.output_tokens)       |add // 0) }
' "$LOG_DIR/events.jsonl" > "$LOG_DIR/usage.json"

jq -c --arg ts "$TS" --arg repo "$REPO" --arg model "$MODEL" \
   '. + {ts:$ts, repo:$repo, model:$model}' "$LOG_DIR/usage.json" \
   >> workspace/dev-knowledge/memory/codex-runs/_ledger.jsonl

Per-run: workspace/tmp/codex-runs/<ts>/usage.json. History: workspace/dev-knowledge/memory/codex-runs/_ledger.jsonl. Optional --max-tokens N budget cap (off by default) SIGTERMs Codex when running output token total exceeds N.

3. Configurable model

# bin/codex-run.sh <REPO> <MODE> <PROMPT_FILE> [MODEL]
MODEL="${4:-${CODEX_DEFAULT_MODEL:-gpt-5-codex}}"
case "$MODEL" in
  gpt-5-codex|gpt-5|o4-mini|o3) ;;
  *) echo "MODEL not allowlisted: $MODEL" >&2; exit 2 ;;
esac
codex exec --model "$MODEL" ...
  • Default in skill config (CODEX_DEFAULT_MODEL), overridable per call.
  • Allowlist prevents typos / accidental use of unavailable or expensive models.
  • Model name recorded in events.jsonl, usage.json, and _ledger.jsonl so cost reporting can group by model.
  • Switching models never relaxes guardrails — every control is model-agnostic.
